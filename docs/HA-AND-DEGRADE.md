# 高可用与演进方向(面试讲述用)

> 这份文档说明:当前项目刻意保持单机/单实例的简单形态,但已为高可用预留了清晰的演进路径。
> 面试被问"怎么保证高可用 / 怎么扩展"时,讲这里的内容。
>
> **核心立场**:当前规模单机够用,过早引入集群/注册中心是过度设计;
> 但我清楚每个组件该怎么演进到高可用,需要时就上。这是"按需演进"而非"不会"。

---

## 一、当前形态(有意为之的简单)

- 应用:article / interaction / user 各单实例
- Redis:单机
- MySQL:单机
- RocketMQ:单 namesrv + 单 broker

这是演示/简历项目的合理形态——能完整跑通核心能力,不被运维复杂度淹没。

---

## 二、各组件的高可用演进路径

### 2.1 应用层:多实例 + 负载均衡
- 现状:每个服务单实例
- 演进:每个服务部署多实例,前面挂负载均衡(Nginx / 云 LB)
- 关键点:**应用要无状态**——本项目应用本身无状态(session 不存本地、缓存在 Redis),
  所以可以直接水平扩展。本地缓存 L1 的一致性已用 MQ 广播解决,多实例没问题。
- 这也是为什么我**没有把状态存在应用内存**的原因之一(点赞缓冲改 Redis Hash 也是为此)

### 2.2 Redis:哨兵 / Cluster
- 现状:单机 Redis
- 演进:
  - **哨兵(Sentinel)模式**:1 主 N 从 + 哨兵监控,主挂自动故障转移。适合中等规模。
  - **Cluster 模式**:数据分片到多节点,容量和吞吐都能扩。适合大规模。
- 代码影响:很小。Redisson 和 Spring Data Redis 都支持配置哨兵/集群地址,业务代码不用改。
- 注意:Cluster 下要注意热点 key 倾斜(单个热 key 仍落单节点),这正是本项目用**本地缓存 L1**
  额外兜热点的原因——Cluster 解决不了单热点,L1 能。

### 2.3 MySQL:主从 + 读写分离
- 现状:单机
- 演进:主从复制,写主库、读从库分担读压力;再大可分库分表
- 代码影响:读写分离可通过中间件(ShardingSphere)或多数据源实现
- 注意:主从有复制延迟,读从库可能读到旧数据,要评估业务能否容忍

### 2.4 RocketMQ:多 master + 主从
- 现状:单 namesrv + 单 broker
- 演进:namesrv 多节点(无状态,挂一个不影响);broker 多 master 多 slave,master 挂了 slave 顶上
- 消息可靠性:同步刷盘 + 主从同步复制(本项目用异步,生产按需调强)

## 三、服务降级(已落地到代码)

高可用不只是"加机器",还包括"依赖挂了我不跟着挂"。本项目代码里实现了几处降级:

### 3.1 user 服务降级(UserClient)
- 文章渲染需要作者名,要调 user 服务。
- user 服务挂了怎么办?**不抛异常,返回兜底名"用户N"**,文章照常展示。
- 原理:依赖隔离——非核心依赖(显示作者名)的故障,不应拖垮核心功能(看文章)。
- 还加了本地缓存(5分钟),既提性能又减少对 user 服务的依赖频率。

### 3.2 点赞服务降级(LikeService)
- 点赞依赖 Redis(幂等、计数)。Redis 异常时怎么办?
- **捕获基础设施异常,返回友好提示"服务繁忙",不向用户暴露技术栈**。
- 理由:点赞不是核心交易,失败可接受,用户重试即可,不影响浏览文章。

### 3.3 缓存重建降级(ArticleCacheManager)
- 缓存击穿重建时抢不到分布式锁,不无限等待,**降级直接查库**。
- 理由:可用性优先,不能让请求因为抢不到锁而无限阻塞。

> 降级的统一思想:**分清核心与非核心,非核心故障要隔离,不能级联拖垮核心链路。**
> 这是微服务容错的基本功,面试高频。

## 四、面试怎么讲高可用(话术)

被问"你这项目怎么保证高可用/怎么扩展":

> "当前是演示形态,都是单机。但架构上为高可用做了准备:
> 应用是无状态的,可以直接多实例水平扩展——这也是我把点赞缓冲从内存改到 Redis 的原因,
> 避免应用持有状态。
> Redis、MySQL、RocketMQ 都有成熟的高可用方案(哨兵/集群、主从、多broker),
> 配置层面就能切换,业务代码基本不动。
> 另外我在代码里做了服务降级:user 服务挂了文章用兜底作者名照常展示,
> Redis 异常时点赞返回友好提示,缓存重建抢不到锁就降级查库——保证非核心依赖的故障不拖垮核心链路。
> 我的原则是:当前规模单机够用,等真有高可用需求时按这个路径演进,不过早引入运维复杂度。"

这段话同时展示了三个层次:
1. 懂高可用的具体方案(哨兵/集群/主从)
2. 懂高可用的前提(应用无状态)
3. 懂容错(服务降级已落地)
而且守住了"按需演进、不过度设计"的一致立场。
